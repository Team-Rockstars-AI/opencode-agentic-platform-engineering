# `/compliance` Skill â€” Regulatory Mapping & Audit Readiness (Terraform Focus)

## Trigger
- Run whenever the operator invokes `/compliance`, requests a regulatory mapping refresh, or needs to prove technical control coverage across Terraform modules and pipelines.
- Activate after `/audit`, `/drift`, or security-checklist runs so their artefacts can be ingested instead of re-scanning.
- This workflow is **read-only**. Never run `terraform apply`, deploy resources, or mutate infrastructure state from this skill.

## Inputs & Evidence Sources
1. **Security baselines:** Latest outputs from `reports/security/security-checklist.md` (generated by the `security-checklist` skill) and `/audit` artefacts under `reports/audit/`.
2. **Drift status:** `DRIFT_REPORT.md` plus any matrices under `artifacts/drift/` to understand deviations between IaC and Azure reality.
3. **IaC + pipelines:**
   - Terraform modules in `modules/terraform/*` and environment scaffolds under `templates/terraform/`.
   - Pipeline definitions in `.github/workflows/*.yml` and `azure-pipelines.yml` (canonical CI) for OIDC proofs.
4. **Regulatory source of truth:** `docs/compliance-mapping-guide.md` â€” use this matrix for binding DORA/BIO/GDPR citations. Never invent mappings outside this guide.

## Steps

1. **Read-only safety check**
   - Confirm no commands in this run will mutate Azure (no `terraform apply`, no `az deployment`, no deleting resources).
   - Export `TF_CLI_ARGS_plan="-lock=false"` if you must open Terraform state for inspection, ensuring you never hold locks accidentally.

2. **Collect prior findings**
   - Import the latest `security-checklist` results and `/audit` report. Normalize findings into `{control_id, severity, remediation, status}`.
   - Parse `DRIFT_REPORT.md` plus `artifacts/drift/*matrix*` to capture controls that are compliant in code but out-of-sync in Azure. Tag any drifted security control as **AT RISK** even if Terraform is correct.

3. **Enumerate Terraform control coverage**
   - Walk `modules/terraform` and `templates/terraform` to list resources that implement each control bucket:
     | Control | What to inspect |
     | --- | --- |
     | **Zero-Secrets Auth (OIDC)** | Pipelines referencing `federatedCredentials`, `azure/login` with `enable-AzPSSession: false`, absence of `client_secret`. Document evidence path (e.g., `.github/workflows/ci.yml`). |
     | **Private Connectivity** | `azurerm_private_endpoint`, `private_service_connection`, `public_network_access_enabled = false` on Key Vault/Storage/App Services. |
     | **NSG Micro-Segmentation** | `azurerm_network_security_group` with default deny rules, subnet associations, NSG flow logs. |
     | **Diagnostic Logging** | `azurerm_monitor_diagnostic_setting`, `log_analytics_workspace_id`, NSG flow log resources. |
     | **Soft Delete & Purge Protection** | `azurerm_key_vault` (`soft_delete_retention_days`, `purge_protection_enabled`), `azurerm_storage_account` (`restore_policy`, `delete_retention_policy`). |
     | **Tagging & CAF Naming** | `locals { default_tags = { ... } }`, `naming` modules, validation blocks enforcing CAF prefixes.
   - Record per-control evidence: file path, Terraform address, whether the control is parameterised per environment, and any gaps.

4. **Static pipeline inspection for OIDC evidence**
   - Review `.github/workflows/` and `azure-pipelines.yml` for `permissions: id-token: write` (GitHub) or `federatedCredentials` (ADO service connections).
   - Ensure no `client_secret`, `service_principal_password`, or stored credentials remain. Flag any secrets block as **NON-COMPLIANT**.

5. **Cross-reference regulatory mappings**
   - For each technical control, join the evidence with `docs/compliance-mapping-guide.md` to pull the authoritative DORA, BIO, and GDPR citations.
   - Build a structured JSON/YAML (e.g., `reports/compliance/<timestamp>-matrix.json`) with fields: `control`, `resource`, `status {pass|fail|at_risk}`, `dora_article`, `bio_control`, `gdpr_article`, `evidence_path`, `notes`.
   - If a control is not implemented anywhere in Terraform, mark it as **FAILED** with remediation referencing the missing module or setting.

6. **Synthesize compliance readiness metrics**
   - Calculate totals: controls evaluated, PASS/FAIL/AT RISK counts, compliance percentage (`pass / total * 100`).
   - Highlight blockers (e.g., missing purge protection) and tie them back to the plan or audit evidence.

7. **Generate `COMPLIANCE_READINESS_REPORT.md`**
   - Instantiate `docs/reports/compliance-readiness-template.md` (or `templates/docs/reports/...` for sample evidence) into `COMPLIANCE_READINESS_REPORT.md` at the repo root or `reports/compliance/`.
   - Populate sections:
     1. **Executive Summary:** overall compliance %, key risks, environments assessed.
     2. **Regulatory Control Matrix:** table mapping Technical Control â†’ Resource/Config â†’ DORA/BIO/GDPR article â†’ Status.
     3. **Non-Compliance Findings:** severity-tagged list with remediation referencing exact Terraform or pipeline files.
     4. **Appendices:** links to `DRIFT_REPORT.md`, `reports/security/security-checklist.md`, `reports/audit/` artefacts, and `docs/compliance-mapping-guide.md` for auditors.

8. **Hand-off to downstream agents**
   - Provide the compliance matrix to `@docs-writer` for archival and to `@security-auditor` for verification of regulatory claims.
   - Notify `@plan-validator` if any AT RISK controls are due to drift, so remediation plans can be prioritized before apply windows.

## Outputs
- `reports/compliance/<timestamp>-compliance-matrix.json|yaml` â€” structured mapping of technical controls to DORA/BIO/GDPR with status and evidence.
- `COMPLIANCE_READINESS_REPORT.md` (or `reports/compliance/COMPLIANCE_READINESS_REPORT.md`) â€” auditor-ready summary referencing the template and mapping guide.
- Cross-links to prior artefacts (`DRIFT_REPORT.md`, security checklist, `/audit` outputs) to prove historical posture without re-running scans.
- Notification payload summarizing PASS/FAIL counts for orchestrator + docs/security agents.
