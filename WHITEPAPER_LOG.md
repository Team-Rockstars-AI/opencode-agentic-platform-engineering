# Platform Engineering — Whitepaper Log

## Milestone: Formalised Reusable Skills Framework (June 2026)

The repository took a significant architectural step by elevating **Skills** from an implicit concept to a first-class, version-controlled, and discoverable abstraction. Prior to this milestone, operational procedures (branch naming, commit hygiene, security review checklists) were scattered across individual agent prompt files — duplicated between root prompts and template prompts, with no single source of truth. The decision was made to extract these into `skills/<name>/SKILL.md` files, each with a standard frontmatter header, a trigger description, a step-by-step procedure, and a defined output format.

Two flagship skills were authored: the `security-checklist` skill, which distils the repository's eight hard-block security criteria into a structured PASS/FAIL/NA checklist with severity-rated finding reports; and the `git-workflow` skill, which enforces consistent branch naming, pre-commit hygiene, commit blacklist rules, and a standardised handoff summary for the test-writer agent. The existing `audit` skill was retrofitted to load `security-checklist` during analysis, and the builder prompts (`builder-infra-tf`, `builder-infra-bicep`, `builder-pipelines`) were updated to delegate workflow rules to the `git-workflow` skill rather than embedding them inline. The template copies under `templates/opencode-config/` were mirrored to match.

A key friction point emerged around the boundary between skills and prompts. Some builder prompts still retain inline security mandates (soft-delete, private connectivity, diagnostic logging) that overlap with the `security-checklist` skill — a deliberate choice to keep implementation directives co-located with builder instructions while reserving the skill for verification. This duplication is acceptable for now but should be reconciled in a future pass. Another concern was the growing structural drift between `.opencode/prompts/` (the master prompts) and `templates/opencode-config/prompts/` (the scaffold outputs). The builder prompts in the master are more detailed and include architecture-review and code-standards skill references not present in the template versions. A follow-up milestone should reconcile these, and also formalise the as-yet-unwritten `code-standards` and `commit-format` skills that are already referenced by name.

## Milestone: Complete Reusable Skills Framework & Prompt Reconciliation (June 2026)

This milestone closed the loop on the skills framework by authoring the six skills that were previously only referenced by name in agent prompts: `code-standards` (91 lines codifying CAF naming, Azure WAF five-pillar alignment, mandatory tagging, and Terraform/Bicep best practices), `commit-format` (92 lines standardising Conventional Commits with nine types, scope rules, and breaking change notation), `architecture-review` (subscription topology, network isolation, central state access, and IAM review criteria), `plan-tracking` (execution plan conversion, resource action taxonomy, and session state maintenance), `doc-standards` (module READMES, ADR format, and runbook guidelines), and `test-patterns` (Terraform 1.6+ .tftest.hcl unit testing with example assertions, plus Bicep validation patterns). Alongside the skill authoring, a critical reconciliation pass aligned the master prompt files under `.opencode/prompts/` with their template counterparts under `templates/opencode-config/prompts/`, eliminating the structural drift that had accumulated. An automated validation script (`scripts/validate-skills.py`) was also introduced to scan all prompt files for broken skill references and gate the CI pipeline if any are found. The net result is a fully self-consistent skills ecosystem where every agent instruction that says "load the X skill" can be traced to an actual `skills/X/SKILL.md` file — a safety and maintainability improvement that eliminates the risk of orphaned references, though the team noted that the `code-standards` skill is dense enough that future splitting into WAF, naming, and tagging sub-skills might improve adoption among builder agents.
