# Platform Engineering — Whitepaper Log

## Milestone: Formalised Reusable Skills Framework (June 2026)

The repository took a significant architectural step by elevating **Skills** from an implicit concept to a first-class, version-controlled, and discoverable abstraction. Prior to this milestone, operational procedures (branch naming, commit hygiene, security review checklists) were scattered across individual agent prompt files — duplicated between root prompts and template prompts, with no single source of truth. The decision was made to extract these into `skills/<name>/SKILL.md` files, each with a standard frontmatter header, a trigger description, a step-by-step procedure, and a defined output format.

Two flagship skills were authored: the `security-checklist` skill, which distils the repository's eight hard-block security criteria into a structured PASS/FAIL/NA checklist with severity-rated finding reports; and the `git-workflow` skill, which enforces consistent branch naming, pre-commit hygiene, commit blacklist rules, and a standardised handoff summary for the test-writer agent. The existing `audit` skill was retrofitted to load `security-checklist` during analysis, and the builder prompts (`builder-infra-tf`, `builder-infra-bicep`, `builder-pipelines`) were updated to delegate workflow rules to the `git-workflow` skill rather than embedding them inline. The template copies under `templates/opencode-config/` were mirrored to match.

A key friction point emerged around the boundary between skills and prompts. Some builder prompts still retain inline security mandates (soft-delete, private connectivity, diagnostic logging) that overlap with the `security-checklist` skill — a deliberate choice to keep implementation directives co-located with builder instructions while reserving the skill for verification. This duplication is acceptable for now but should be reconciled in a future pass. Another concern was the growing structural drift between `.opencode/prompts/` (the master prompts) and `templates/opencode-config/prompts/` (the scaffold outputs). The builder prompts in the master are more detailed and include architecture-review and code-standards skill references not present in the template versions. A follow-up milestone should reconcile these, and also formalise the as-yet-unwritten `code-standards` and `commit-format` skills that are already referenced by name.

## Milestone: Complete Reusable Skills Framework & Prompt Reconciliation (June 2026)

This milestone closed the loop on the skills framework by authoring the six skills that were previously only referenced by name in agent prompts: `code-standards` (91 lines codifying CAF naming, Azure WAF five-pillar alignment, mandatory tagging, and Terraform/Bicep best practices), `commit-format` (92 lines standardising Conventional Commits with nine types, scope rules, and breaking change notation), `architecture-review` (subscription topology, network isolation, central state access, and IAM review criteria), `plan-tracking` (execution plan conversion, resource action taxonomy, and session state maintenance), `doc-standards` (module READMES, ADR format, and runbook guidelines), and `test-patterns` (Terraform 1.6+ .tftest.hcl unit testing with example assertions, plus Bicep validation patterns). Alongside the skill authoring, a critical reconciliation pass aligned the master prompt files under `.opencode/prompts/` with their template counterparts under `templates/opencode-config/prompts/`, eliminating the structural drift that had accumulated. An automated validation script (`scripts/validate-skills.py`) was also introduced to scan all prompt files for broken skill references and gate the CI pipeline if any are found. The net result is a fully self-consistent skills ecosystem where every agent instruction that says "load the X skill" can be traced to an actual `skills/X/SKILL.md` file — a safety and maintainability improvement that eliminates the risk of orphaned references, though the team noted that the `code-standards` skill is dense enough that future splitting into WAF, naming, and tagging sub-skills might improve adoption among builder agents.

## Milestone: Security Remediation & Hardening (June 2026)

This milestone shifted focus from the meta-layer of agent skills back to the infrastructure modules themselves, applying five targeted security hardening fixes driven by the security checklist criteria codified in the earlier milestones. The first and most impactful decision was to replace the `Contributor` role assignment in the OIDC bootstrap module with a custom, least-privilege role definition (`custom-pipeline-deployer-*`). The original design used `Contributor` for expediency during early development, but this violated the principle of least privilege and exposed the subscription to excessive permissions from pipeline credentials. The custom role was scoped to precisely the actions the pipeline legitimately needs — resource group management, network resource CRUD, Key Vault and Storage Account deployment, Container Apps and App Service management, Managed Identity and federated credential operations, and diagnostic/Log Analytics configuration — while excluding everything else. Both the Terraform and Bicep variants were updated in lockstep to maintain framework parity.

The second decision targeted Key Vault security posture. The existing Key Vault module was hardened with three complementary controls: purge protection was enabled to prevent permanent deletion of secrets and certificates; public network access was disabled to force all client traffic through the private network; and network ACLs were configured with a default-deny posture that only bypasses for trusted Azure platform services. These three together mean that a compromised pipeline credential cannot exfiltrate secrets over the public internet, and accidental vault deletion can be recovered within the soft-delete retention window. The soft-delete retention period itself was set to 7 days (Terraform as a variable, Bicep with a decorator-validated default) as a pragmatic balance between recovery window and operational hygiene.

The third architectural decision was the introduction of subnet-level micro-segmentation through Network Security Groups in the network baseline module. Previously, the virtual network and subnets were deployed without any NSG attachments, meaning intra-VNet traffic was unrestricted — any workload could reach any other subnet. The milestone introduced four NSGs (workloads, appgw, endpoints, and runners for enterprise deployments) and associated each to its respective subnet. A concrete security rule was authored on the endpoints NSG to restrict inbound HTTPS traffic exclusively from the workloads subnet prefix, enforcing the principle that only application workloads should reach Private Endpoint resources. The remaining NSGs were deployed without rules, ready for consumers to add application-specific restrictions, but their very presence blocks all inbound traffic by default (since NSGs without explicit allow rules deny all inbound traffic not matching a rule).

The fourth fix was a pipeline resilience hardening pass: every script block in both the GitHub Actions and Azure DevOps pipeline templates was audited and standardised to use `set -euo pipefail`. This ensures that any intermediate command failure, unset variable reference, or pipe failure immediately halts the pipeline rather than silently continuing with a partial or corrupted deployment state. While most blocks already had this, the audit revealed a few gaps and inconsistencies that were rectified.

The fifth and final decision was to make Application Gateway HTTPS support configurable rather than hardcoded. The network baseline module already deployed a WAF_v2 Application Gateway with an HTTP listener only. The milestone added a `ssl_certificate_key_vault_secret_id` input variable and used dynamic blocks (Terraform) and conditional expressions (Bicep) to optionally provision an HTTPS frontend port (443), HTTPS listener, SSL certificate binding sourced from Key Vault, and a second request routing rule. The HTTPS listener and routing rule are only created when the variable is provided, avoiding unnecessary complexity for environments that do not require TLS termination at the gateway. The HTTP listener is preserved alongside it, allowing both protocols to coexist. A notable friction point is that the SSL certificate must already exist in Key Vault — the module does not handle certificate provisioning or renewal, placing that responsibility on the operator.

Taken together, these five changes represent a significant improvement in the platform's security baseline without incurring additional Azure costs (NSGs are free, custom RBAC roles are free, and the Application Gateway WAF_v2 meter is unchanged by additional listeners). The work was split across the Terraform and Bicep builder agents, each implementing the changes in their respective module trees while maintaining interface parity through shared variable schemas.

## Milestone: Optimal Platform Hardening & Micro-segmentation (June 2026)

This milestone achieved a comprehensive, zero-trust security posture across all infrastructure modules and templates, resolving all remaining compliance findings. The first major decision was to implement strict, subnet-level micro-segmentation by adding explicit `DenyAllInbound` rules (priority 4000) to all Network Security Groups (workloads, appgw, endpoints, and runners). To ensure legitimate traffic still flows, explicit inbound allow rules were configured to authorize only Internet-to-AppGateway, AppGateway-to-Workloads, and Workloads-to-Endpoints traffic on required ports (80/443), creating a robust, multi-layered defense-in-depth network architecture. The second decision was to refine the custom pipeline deployer role by removing the `listKeys/action` permission, thereby enforcing strict Microsoft Entra ID data-plane authentication for all storage operations and eliminating access key exposure risks.

The third decision focused on comprehensive input validation and parameter decorators. Robust `validation {}` blocks were added to all remaining Terraform variables (including `project_name`, `location`, `resource_group_name`, and `tenant_id`), and matching `@minLength`, `@maxLength`, `@allowed`, and `@secure` decorators were applied to all Bicep parameters across the network baseline, OIDC bootstrap, and private runner modules. This guarantees compile-time and plan-time input safety across both IaC frameworks. The fourth decision was to wire up central diagnostic settings for all network resources (Application Gateway and all four NSGs) and runner resources (Container App Environment and Container App) to stream logs and metrics to the central Log Analytics workspace, eliminating critical audit trail blind spots. Finally, the Key Vault soft-delete retention period was increased from the 7-day minimum to 90 days across all modules and templates to align with the Microsoft Cloud Security Benchmark and maximize data recovery windows.
