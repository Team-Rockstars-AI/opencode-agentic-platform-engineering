# `/compliance` Skill — Scaffolded Repo Instructions

## Trigger
- Execute whenever the platform team in a scaffolded repo runs `/compliance` or requests an updated `COMPLIANCE_READINESS_REPORT.md`.
- Always run after refreshed `/audit`, `/drift`, and `security-checklist` outputs exist to avoid duplicative scans.
- This skill is **read-only**. Never run `terraform apply`, `az deployment`, or modify Azure resources from `/compliance`.

## Evidence Inputs
1. **Security artefacts:**
   - `reports/security/security-checklist.md` (latest checklist pass/fail table).
   - `/audit` outputs under `reports/audit/`.
2. **Drift artefacts:** `DRIFT_REPORT.md` plus any `reports/drift/*.matrix.json` generated by the `/drift` command.
3. **Infrastructure-as-Code:**
   - Terraform environments under `terraform/` and reusable modules under `modules/terraform/` (if the scaffold included them).
   - Pipeline files inside `.github/workflows/` and/or `pipelines/azure-devops/` (depending on chosen DevOps host).
4. **Mapping reference:** `docs/compliance-mapping-guide.md` (copied from the template) — the only approved source for DORA/BIO/GDPR mappings.

## Steps

1. **Confirm read-only posture**
   - Explicitly note in the run log that `/compliance` does not gate any deployment or mutate state.
   - Use `terraform show -json` or static file parsing only; do not acquire backend locks or push plans to Azure.

2. **Ingest prior reports**
   - Load the checklist, audit report, and drift matrix into a working data structure (JSON/YAML). Normalize statuses (`PASS`, `FAIL`, `AT_RISK`).
   - Tag any drifted control (e.g., Key Vault missing purge protection in Azure) as **AT_RISK** even if Terraform already enforces the control.

3. **Scan Terraform + pipelines for control implementation**
   - From repo root, enumerate controls and evidence:
     | Control | Evidence to capture |
     | --- | --- |
     | **Zero-Secrets Auth (OIDC)** | `.github/workflows/*.yml` `permissions.id-token: write`, Azure DevOps service connections using federated credentials, absence of `client_secret` fields. |
     | **Private Connectivity** | Terraform resources `azurerm_private_endpoint`, `public_network_access_enabled = false` on Key Vault/Storage/Container Apps. |
     | **NSG Micro-Segmentation** | `azurerm_network_security_group` rules + subnet associations enforcing default deny. |
     | **Diagnostic Logging** | `azurerm_monitor_diagnostic_setting`, NSG flow logs, Log Analytics workspace IDs. |
     | **Soft Delete & Purge Protection** | Key Vault retention, Storage account soft delete, SQL short-term retention. |
     | **Tagging & CAF Naming** | `locals { default_tags = { ... } }`, naming modules, Terraform variable validation enforcing CAF prefixes. |
   - Note file paths (e.g., `terraform/envs/prod/key-vault.tf`) for each control.

4. **Reference regulatory mappings**
   - For every control, look up the DORA, BIO, and GDPR columns in `docs/compliance-mapping-guide.md`.
   - Build `reports/compliance/<timestamp>-matrix.json` entries: `{control, resource, status, dora_article, bio_control, gdpr_article, evidence_path, notes}`.

5. **Summarize compliance posture**
   - Compute counts for PASS / FAIL / AT_RISK and overall compliance %.
   - Flag CRITICAL failures (e.g., purge protection disabled) and tie them to regulatory impact.

6. **Generate the report**
   - Instantiate `templates/docs/reports/compliance-readiness-template.md` into `reports/compliance/COMPLIANCE_READINESS_REPORT.md` (create the folder if needed).
   - Fill in: executive summary, regulatory matrix, and non-compliance findings (severity, impacted control, remediation referencing Terraform/pipeline files).
   - Link supporting artefacts: `DRIFT_REPORT.md`, `reports/security/security-checklist.md`, `reports/audit/`, and the mapping guide.

7. **Distribute outputs**
   - Provide the matrix + report to `@docs-writer` for archival and to `@security-auditor` for validation.
   - Surface any FAIL/AT_RISK controls to `@orchestrator` so follow-up `/expand` or remediation tasks can be scheduled.

## Outputs
- `reports/compliance/<timestamp>-matrix.json|yaml` — status of each technical control with DORA/BIO/GDPR references and evidence.
- `reports/compliance/COMPLIANCE_READINESS_REPORT.md` — populated from the template with executive summary, matrix, and findings.
- Optional notifications (issue, PR comment, or chat message) summarizing PASS/FAIL counts for leadership visibility.
